# LLM 术语解释器（文科生版）

## 目标

把抽象技术名词翻译成文科生可理解、可复述、可迁移的认知结构。
结合两类优势：
- 概念拆解优势：采用 8 维概念解剖法，避免只讲定义不讲来龙去脉。
- 行文叙事优势：采用“问题驱动 + 第一人称 + 生活比喻 + 辩证反思”的写法，降低阅读门槛。

## 设计锚点
- 拆解框架来源：`lijigang/ljg-explain-concept` 的“多维概念解剖”思想。
- 文风来源：公众号技术概念长文常用的“个人经历开场 + 问题推进 + 比喻解释 + 反思收束”结构。

## 快速使用（自动生成 Markdown）
先生成固定栏目草稿，再填充内容：

```bash
python3 .agents/skills/llm-term-explainer/scripts/generate_concept_markdown.py token
```

默认输出到：`output/explainer/token-concept.md`

常用参数：
- 指定输出文件：
```bash
python3 .agents/skills/llm-term-explainer/scripts/generate_concept_markdown.py embedding -o output/explainer/embedding-concept.md
```
- 生成短版卡片（4 段）：
```bash
python3 .agents/skills/llm-term-explainer/scripts/generate_concept_markdown.py mha --short
```
- 覆盖已有文件：
```bash
python3 .agents/skills/llm-term-explainer/scripts/generate_concept_markdown.py token --force
```

## 执行流程

### 步骤 1：锁定受众和深度
先在心里确认默认受众：不懂技术的文科生。
除非用户明确要求专业版本，否则遵守以下深度限制：
- 单段不超过 4 个新术语。
- 每个新术语第一次出现时立即给白话解释。
- 数学符号只作为补充，不作为主线。

### 步骤 2：概念预处理
对目标术语先做 4 个判断：
1. 这个概念在 LLM 管线里属于哪一层（输入、表示、计算、输出、训练、应用）。
2. 它解决了什么旧问题（没有它会怎样）。
3. 它最容易和什么概念混淆。
4. 哪个生活场景最适合做核心比喻。

### 步骤 3：按 8 维拆解（核心）
按顺序完成以下 8 维，每一维至少给 2-3 句，不要只写标题：

1. 历史溯源
- 这个概念在什么痛点下出现。
- 旧方案是什么，卡在哪里。

2. 辩证分析
- 它带来的收益是什么。
- 它引入的代价或副作用是什么。

3. 现象还原
- 只描述“它看起来怎么工作”，先不推公式。
- 用“你在实际使用 AI 时会看到什么现象”来落地。

4. 语言拆解
- 解释术语字面含义、常见中文译法、词源线索。
- 说明名字为什么容易误导。

5. 机制最小化
- 用最少步骤讲清机制（通常 3 步）。
- 可给 1 条极简公式，但必须晚于直觉解释。

6. 关系网络
- 说明它在整体中的前后依赖。
- 至少比较 2 个易混淆概念（如 token vs word）。

7. 内观视角
- 用第一人称写 4-8 行独白：“如果我是这个概念……”
- 目的是让读者形成记忆钩子。

8. 顿悟压缩
- `The One` 公式：`概念 = 输入 -> 核心变换 -> 输出`
- 费曼一句话：20-35 字。
- ASCII 拓扑图：只画关键节点和方向。

### 步骤 4：按叙事文风落稿
落稿时遵守以下顺序：
1. 开篇三板斧：引用锚点 -> 第一人称情境 -> 结构预告。
2. 每一节以问题句开头，优先使用“你有没有想过……”
3. 先讲故事和比喻，再讲机制和术语。
4. 每写完 2 节，插入一句“前文回顾”做连接。
5. 在结尾明确“这个比喻哪里不成立”，避免误导。

## 输出模板（默认完整版）

```markdown
# 【概念名】：一句话定义

> 引用锚点：一句权威观点（可意译）

作者导语：第一人称个人困惑/经历（2-4句）
导读：本文先...再...最后...

---

## 1. 你有没有想过
问题引入 + 生活场景

## 2. 这个概念从哪来
背景痛点 -> 旧方案 -> 新突破 -> 名称来源

## 3. 直觉建立
核心比喻 + 对应关系

## 4. 原理最小集
用 3 步解释机制（必要时给极简公式）

## 5. 如果没有它
旧方案、优势、代价、未来替代

## 6. 比喻的边界
比喻遮蔽了什么、何时失效

## 7. 概念关系图
与相关概念的区别与连接

## 8. 如果我是这个概念
第一人称独白（4-8行）

## 9. 顿悟压缩
### The One 公式
### 一句话记住它
### ASCII 拓扑图
```

## 快速模式（用户只要短回答时）
如果用户明确要求“简短点”，输出 4 段：
1. 一句话定义
2. 一个生活比喻
3. 一个易混淆对比
4. 一句代价提醒

## 术语准确性底线
写作时不要违反这些底线：
- 不把 token 说成“词”；应说“可计算切分单元”。
- 不把 embedding 说成“词典释义”；应说“语义向量表示”。
- 不把 attention 说成“只看一个词”；应强调“对上下文加权”。
- 不把 MHA 说成“重复同一件事”；应强调“多子空间并行关注”。
- 不把 RAG 说成“模型记住了新知识”；应强调“检索增强，不等于参数更新”。

具体术语口径参考：`references/llm-atom-concepts.md`

## 禁止事项
- 禁止上来堆公式或英文缩写。
- 禁止全程第三人称科普腔，必须有“我”的叙事视角。
- 禁止只讲优点不讲代价。
- 禁止比喻后不指出边界。
- 禁止结尾没有压缩记忆点（公式 + 一句话 + ASCII 图）。
