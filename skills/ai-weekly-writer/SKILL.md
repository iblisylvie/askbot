# AI 技术周报生成器

根据每日新闻原件（原始爬取内容），先生成每日日报，再汇总生成面向 AI 工程师和转型者的技术周报。

## 工作流程概览

```
每日新闻原件 (output/YYYY-MM-DD/) 
    ↓ 步骤1: 生成每日日报
每日日报 (AI科技日报_YYYY-MM-DD_content.md)
    ↓ 步骤2: 汇总生成周报
AI技术周报
```

## 输入格式

每日新闻原件存储在 `output/YYYY-MM-DD/` 目录下：
- `summary.csv` - 文章元数据（标题、公众号、摘要、链接、本地文件路径）
- `文章标题.html` - 文章内容HTML文件（原始爬取）
- `文章标题.md` - 文章内容Markdown文件（HTML转换后，用于降低处理成本）

### summary.csv 格式示例

```csv
account_name,app_msg_id,digest,link,title,create_time,local_file_path
机器之心,2651012094,真正让一个人活成一支队伍。,http://mp.weixin.qq.com/s?...,实测夸克「千问划词快捷指令」...,2026-01-15 03:48:47,output/2026-01-15/实测夸克「千问划词快捷指令」....html
```

## 步骤1: 生成每日日报

对于每一天的原始新闻数据，生成结构化的日报。

### 日报格式 (AI科技日报_YYYY-MM-DD_content.md)

```markdown
# AI科技日报 - YYYY年MM月DD日

## 今日精选文章

### 1. [文章标题]
- **来源**: 公众号名称
- **摘要**: 文章摘要
- **链接**: 文章链接
- **核心内容**: 从Markdown内容中提取的关键信息

...

## 今日热点标签
- 标签1, 标签2, 标签3...

## 主编简评
今日AI领域的主要动态一句话总结...
```

### 生成日报的步骤

1. 读取 summary.csv 获取当日所有文章元数据
2. **HTML 转 Markdown（降低成本）**
   - 对于每篇文章的 HTML 文件，使用 `html2md.js` 转换为 Markdown
   - 转换后的 Markdown 文件保存在同目录，扩展名为 `.md`
   - 转换脚本会自动清理微信 UI 元素、JavaScript 代码等无用内容
   - 使用 Markdown 替代原始 HTML 可显著降低 token 消耗
3. 读取每篇文章的 **Markdown** 内容，提取核心信息
4. 按重要性/热度排序文章
5. 生成结构化日报文件到 `output/AI科技日报_YYYY-MM-DD_content.md`

### HTML 转 Markdown 工具使用

**脚本位置：** `$AI_WEEKLY_WRITER_HOME/scripts/html2md.js`

**环境变量设置（推荐）：**
```bash
# 添加到 ~/.bashrc 或 ~/.zshrc，使所有 agent 可用
export AI_WEEKLY_WRITER_HOME="$HOME/.claude/skills/ai-weekly-writer"
# 其他 agent 可能使用不同路径，如：
# export AI_WEEKLY_WRITER_HOME="$HOME/.agents/skills/ai-weekly-writer"
```

**首次使用：**
```bash
cd $AI_WEEKLY_WRITER_HOME
npm install
```

**使用示例：**
```bash
# 转换单个 HTML 文件
node $AI_WEEKLY_WRITER_HOME/scripts/html2md.js output/2026-01-15/文章标题.html
# 输出：同目录下的 文章标题.md

# 批量转换某天所有文章
for f in output/2026-01-15/*.html; do
    node $AI_WEEKLY_WRITER_HOME/scripts/html2md.js "$f"
done
```

**html2md.js 脚本功能：**
- 自动移除 `<script>`、`<style>`、`<head>` 等无用标签
- 清理微信 UI 元素（"微信扫一扫"、"继续滑动"等）
- 保留文章正文、图片、链接等核心内容
- 将 HTML 转换为干净的 Markdown 格式

**注意：** 脚本依赖相对路径的 `node_modules`，请确保在 `$AI_WEEKLY_WRITER_HOME` 目录下运行 `npm install`。

## 内容筛选标准

### 头条新闻选择（关键步骤）

**头条数量：2-3 条，必须覆盖本周最重大的发布**

优先选择以下类型，按优先级排序：

1. **大厂新模型发布**（必须检查，禁止遗漏）
   
   **国外大厂：**
   - OpenAI（GPT 系列、Codex、o3）
   - Anthropic（Claude 系列、Opus/Sonnet）
   - Google（Gemini 系列、Genie 等世界模型）
   - Meta（Llama 系列）
   - xAI（Grok 系列）
   
   **国内大厂（重点，容易遗漏）：**
   - **智谱AI**（GLM 系列、ChatGLM、GLM-OCR、GLM-5）
   - **MiniMax**（abab 系列、M2、M2.5、海螺AI）
   - **字节跳动**（豆包、Seedance、云雀）
   - **DeepSeek**（DeepSeek-V 系列、DeepSeek-R1）
   - **月之暗面**（Kimi、K2.5）
   - **百川智能**（Baichuan 系列）
   - **零一万物**（Yi 系列）
   - **阶跃星辰**（Step 系列）
   - **面壁智能**（MiniCPM、SALA架构）
   - **阿里**（通义千问、Qwen 系列）
   - **百度**（文心一言、Ernie）
   
   **检查方式（系统性扫描）：**
   - 在输入数据中搜索以下关键词：
     - 国外："Opus"、"Sonnet"、"Claude 4"、"GPT-5"、"GPT 5"、"Codex"、"Gemini"、"Genie"、"Llama 3"、"Grok"
     - 国内："GLM-5"、"GLM-OCR"、"MiniMax"、"M2.5"、"M2"、"豆包"、"DeepSeek"、"Kimi K2"、"Kimi K3"、"Baichuan"、"Yi-"、"Step-"、"通义千问"、"文心"
   - **必须**：遍历所有summary.csv，检查是否有模型发布相关文章

2. **重大产品发布**
   - AI 编程工具（如 Codex App、Cursor 重大更新）
   - AI Agent 平台/框架
   - 世界模型/多模态模型

3. **技术范式变化**（新架构、新训练方法、新评估标准）

4. **开源重磅项目**（高 Star 增长、解决关键痛点）

**头条选择原则：**
- 如果有多条同类型新闻（如同时有 Opus 4.6 和 GPT-5.3 发布），优先考虑直接竞争/对比关系的内容
- 头条标题必须点明核心变化，使用动词开头

避免：
- 纯融资新闻（除非金额特别巨大或有技术背景）
- 重复性的产品更新
- 过于垂直的硬件新闻（除非有软件突破）

### 工具/项目选择

每个工具必须回答：
- **工程师能直接用吗？** 优先选有代码/可试用的
- **解决什么问题？** 明确的问题定义
- **有学习价值吗？** 看代码能学到东西

优先级：
1. GitHub 开源项目（有实际代码）
2. 可在线试用的工具（有 demo）
3. 有技术博客详细解读的新项目

### 学习资源选择

优先：
- 高质量技术博客/教程
- 重要论文的通俗解读
- 实战项目/代码示例

避免：
- 纯营销性质的内容
- 过于基础的入门教程（除非有独特角度）

## 写作风格指南

### 语言风格

- **简洁直接**：避免废话，每句话传递信息
- **技术但不晦涩**：用工程师熟悉的术语，但解释清楚背景
- **实用导向**：每段内容都回答"这对我有什么用"
- **人格化**：适当加入主观判断，建立信任感

### 标题写作

- 头条标题：动词开头，点明核心变化
  - 好：OpenAI 发布 Operator，AI 开始帮你操作浏览器
  - 差：OpenAI 发布新产品

- 工具名称：保留英文原名，可加中文解释
  - 例：Browser Use（浏览器自动化框架）

### 格式规范

- 使用中英文混排时，英文前后加空格
- 链接统一放在文末或表格中
- 代码/命令用行内代码块标注
- 重要信息用 **加粗** 标注

## 链接处理规则

### 1. 微信链接处理（mp.weixin.qq.com）

**问题：** 微信链接在微信外部访问时可能显示 "parameter error"

**处理方式：**
- 如果原文提供的是非微信链接（如官网、GitHub、技术博客），优先使用非微信链接
- 如果只有微信链接，标注 "（需在微信内打开）"
- 如果链接不完整或已过期，标注 "原文未提供有效外部链接"

**示例：**
```markdown
❌ - **链接：** http://mp.weixin.qq.com/s?__biz=...
✅ - **链接：** https://example.com/article （需在微信内打开，或搜索"文章标题"）
```

### 2. 禁止编造链接

- **必须**：只使用原文提供的链接，或你自己验证过的真实链接
- **禁止**：基于命名规则推测链接（如 `github.com/公司名/项目名`）

**处理方式：**
- 如果原文未提供链接：写"原文未提供链接"或删除该条目
- 如果不确定链接是否正确：标注"链接待确认"

### 3. 外部链接来源标注（统一格式）

**所有来源必须统一使用以下格式：**

```
（来源：XXX，YYYY-MM-DD）
```

**具体规范：**

| 场景 | 正确格式 | 错误格式 |
|------|----------|----------|
| 工具/项目链接 | `https://example.com（来源：Z Potentials，2026-02-23）` | `链接：https://example.com` |
| 无外部链接 | `原文未提供外部链接（来源：阶跃星辰官方公众号，2026-02-27）` | `链接：待确认` |
| 学习资源来源 | `- **来源：** 智谱AI官方公众号，2026-02-22` | `- **来源：** 智谱AI` |
| 彩蛋/案例 | `...解决方案（来源：新智元，2026-02-23）` | `...解决方案` |

**来源名称规范：**
- 官方公众号：使用"XXX官方公众号"格式，如"阶跃星辰官方公众号"
- 第三方媒体：直接使用媒体名称，如"Z Potentials"、"新智元"
- 个人/访谈：使用"整理自XXX访谈"，如"整理自OpenAI API工程负责人访谈"

**日期获取方式：** 提取链接所在文章的本地文件路径中的日期文件夹，如 `output/2026-02-18/文章标题.html` → 日期为 `2026-02-18`

**示例：**
```markdown
| **链接** | https://happycapy.ai/（来源：Z Potentials，2026-02-23） |

- **来源：** 智谱AI官方公众号（2026-02-22）

阿里云AI为冬奥赛事提供智能转播解决方案（来源：新智元，2026-02-23）
```

## 准确性要求（关键）

### 1. 严格遵循原文，禁止夸大、禁止编造

- **必须**：准确转述原文的限定词（"部分"、"某些"、"在特定场景下"等）
- **禁止**：为了可读性简化掉关键限定，导致事实失真
- **禁止编造技术细节**：如优化方式（TPU/GPU/量化）、具体架构、未提及的性能数据等

**错误示例：**
> 原文："支持1M token上下文窗口（beta），长上下文处理能力较前代提升四倍"  
> ❌ 错误："TPU 优化加速：推理速度显著提升"（原文未提及 TPU 或推理速度优化）  
> ✅ 正确："支持1M token上下文窗口（beta），长上下文处理能力较前代提升四倍"

**错误示例 1 - 夸大：**
> 原文："部分性能指标超越 Genie 3"  
> ❌ 错误："性能超越谷歌闭源的 Genie 3"（省略"部分指标"导致夸大）  
> ✅ 正确："部分性能指标超越 Genie 3"

**错误示例 2 - 数据归属错误：**
> 原文："METR报告显示GPT-5.2在软件任务中实现约6.6小时50%时间跨度表现"  
> ❌ 错误：将这段描述放在 GPT-5.3 下，写为"GPT-5.3 与前代 GPT-5.2 相比，实现约 6.6 小时 50% 时间跨度表现"（数据归属错误，6.6小时是 GPT-5.2 自身的表现，不是 GPT-5.3 的对比数据）  
> ✅ 正确：明确标注这是 GPT-5.2 的数据，或单独说明 GPT-5.3 的实际性能（如"25% 更快推理，2.09 倍 token 效率"）

**错误示例 2.5 - 编造链接（严重错误）：**
> 场景：原文只提到"阶跃星辰核心团队做客Reddit r/LocalLLaMA：Step 3.5 Flash 的工程取舍与下一步"，未提供任何外部链接  
> ❌ 错误：在工具介绍中添加`https://github.com/stepfun-ai/Step-3.5-Flash`（这是根据命名规则推测的链接，原文未提供）  
> ✅ 正确：链接栏写"原文未提供外部链接"或"来源：阶跃星辰官方公众号，2026-02-27"

**错误示例 3 - 遗漏限定词导致事实失真：**
> 原文："2025年OpenAI调整后毛利率从预期的46%降至33%"  
> ❌ 错误："毛利率从46%降至33%"（遗漏"预期的"限定词，让读者误以为是实际数据从46%降到33%，而实际上可能是预期目标未达成）  
> ✅ 正确："毛利率从预期的46%降至33%"或"毛利率为33%，低于此前预期的46%"

**错误示例 4 - 笼统泛化，丢失核心技术细节：**
> 原文："这一方案基本消除了KV缓存的I/O开销"（来自：DeepSeek开源智能体推理神器，吞吐量平均提升近2倍，北大实习生立功）  
> ❌ 错误："智能体推理性能优化工具包"（过于笼统，丢失了KV缓存、I/O优化等核心技术细节）  
> ✅ 正确："DeepSeek开源的KV缓存I/O优化方案，吞吐量平均提升近2倍，北大实习生主导开发"

**错误示例 5 - 产品类型误判：**
> 原文："阶跃星辰核心团队做客Reddit r/LocalLLaMA：Step 3.5 Flash 的工程取舍与下一步"，摘要："从 11B 激活参数到 350 TPS 推理"  
> ❌ 错误："Step 3.5 Flash是阶跃星辰的高效推理技术"（把模型误判为技术）  
> ✅ 正确："Step 3.5 Flash是阶跃星辰发布的大语言模型，11B激活参数，350 TPS推理速度"

**错误示例 6 - 来源类型误判+来源格式不统一：**
> 原文标题："深度｜OpenAI API华人工程负责人：模型会把你的脚手架当早餐吃掉..."（来自Z Potentials公众号的访谈文章，2026-02-24）  
> ❌ 错误1：来源标注为"OpenAI API华人工程负责人"，类型标注为"工程实践"（让人误以为是官方指南）  
> ❌ 错误2：未标注日期  
> ❌ 错误3：格式不统一，有些写"来源：XXX"，有些只写"XXX"  
> ✅ 正确：类型标注为"深度访谈"，来源标注为"Z Potentials（2026-02-24）"（统一格式：`（来源：XXX，YYYY-MM-DD）`）

### 2. 信息必须可验证、来源必须准确

- **必须**：每条信息都能在原文中找到依据
- **禁止**：为了凑数保留无法验证的信息
- **来源标注必须准确**：区分原始发布方 vs 转载/整理方

**来源标注示例：**
> 原文：DataFunTalk 公众号文章，标注"整理自 OpenAI Build Hour 技术分享"  
> ❌ 错误：来源标注为 "OpenAI"（让人误以为是官方发布）  
> ✅ 正确：来源标注为 "DataFunTalk（整理自 OpenAI Build Hour 技术分享）"

**学习资源处理：**
- 如果原文提到某资源但未提供获取方式 → 删除该条目或写"原文未提供获取链接"
- 不要写"可通过相关公众号获取"这类模糊表述

### 3. 融资状态必须准确

融资新闻必须准确描述状态，禁止混淆：

| 状态 | 正确表述 | 错误表述 |
|------|----------|----------|
| 拟融资/谈判中 | "拟筹集"、"计划融资"、"谈判中" | ❌ "完成融资"、"获得融资" |
| 官宣但未交割 | "宣布完成"、"官宣" | ❌ 不提状态，默认已完成 |
| 实际交割 | "已完成"、"交割完毕" | - |

**示例：**
- 原文："OpenAI正寻求筹集高达1000亿美元资金...谈判中...尚未最终确定"
- ❌ 错误："OpenAI的千亿美元融资"（让人误解已完成）
- ✅ 正确："OpenAI拟筹集最高1000亿美元"

### 4. 企业案例必须可溯源

- **必须**：企业案例必须来自原文，并能在原文中找到确切依据
- **禁止**：不能编造企业案例，即使看起来合理
- **标注**：如果案例来自原文，不需要特别标注；如果是常识性内容可以保留

**示例：**
- 原文提到"跨境魔方AI外贸员...筛选100家客户时间从3天压缩至5分钟"
- ✅ 正确：直接引用原文案例
- ❌ 错误：把"跨境魔方"改成"某跨境电商公司"（模糊化后看起来像编的）

### 5. 常见错误案例（务必警惕）

以下是从历史周报中总结的**高频错误类型**，生成时务必警惕：

#### 错误类型1：单位/数量级错误
| 错误 | 正确 | 核查方法 |
|------|------|----------|
| "1700亿美元" | "250亿美元（约1700亿人民币）" | **看到数字必须找原文**，确认货币单位 |
| "10亿用户" | "10亿参数" / "1亿用户" | 数字+名词组合必须精确匹配原文 |

**根因**：被文章标题省略单位误导，或脑补了常见单位。

#### 错误类型2：产品归属错误
| 错误 | 正确 | 核查方法 |
|------|------|----------|
| "Anthropic推出OpenClaw" | "奥地利独立开发者Peter Steinberger开发OpenClaw" | **看到产品名必须找原文**，确认开发主体 |
| "OpenAI发布Claude" | "Anthropic发布Claude" | 大厂vs社区项目、A公司vs B公司必须分清 |

**根因**：混淆官方产品 vs 社区开源项目，或张冠李戴。

#### 错误类型3：融资/上市状态错误
| 错误 | 正确 | 核查方法 |
|------|------|----------|
| "MiniMax递交招股书" | "MiniMax发布上市后首份财报（00100.HK）" | **股票代码是上市标志**，看到"首份"要确认修饰对象 |
| "拟融资1000万" | "已完成1000万融资" | 区分"拟"/"官宣"/"完成"三种状态 |

**根因**：被"首份"、"财报"、"招股书"等词汇混淆时间线。

#### 错误类型4：修饰对象错误
| 错误 | 正确 | 核查方法 |
|------|------|----------|
| "首份招股书" | "首份财报" | **限定词修饰对象必须明确** |
| "预期毛利率46%降至33%" | "预期毛利率从预期的46%降至33%" | 保留原文限定词"预期的" |

**根因**：忽略了限定词（如"预期的"、"约"、"首份"）的精确修饰对象。

- [ ] **产品归属核查（必做）：**每个产品/工具必须确认开发主体（大厂官方 vs 社区开发者），禁止张冠李戴

## 完整工作流程

### 场景：用户指定日期范围生成周报

**输入示例**：
> 我有每日新闻原件，存储路径为：output/xxxx-xx-xx/，请帮我为 2026-01-15 到 2026-02-08 这周生成周报

**处理步骤：**

1. **确认数据可用性**
   - 检查指定日期范围内每一天的 `output/YYYY-MM-DD/summary.csv` 是否存在
   - 如果有缺失日期，告知用户并询问如何处理

2. **生成每日日报**（对于每一天）
   - 读取 `output/YYYY-MM-DD/summary.csv` 获取文章列表
   - **HTML 转 Markdown**：使用 `scripts/html2md.js` 将每篇文章的 HTML 转换为 Markdown
   - 读取转换后的 **Markdown** 内容，提取核心信息
   - 过滤广告/低质量内容
   - 按重要性排序
   - 生成 `output/AI科技日报_YYYY-MM-DD_content.md`

3. **检查大厂新模型发布**（关键步骤，禁止遗漏）
   
   **系统性扫描流程：**
   
   a. **遍历所有summary.csv文件**，搜索以下关键词（国内大厂重点关注）：
      - 国外模型："Opus"、"Sonnet"、"Claude 4"、"GPT-5"、"GPT 5"、"Codex"、"Gemini"、"Gemini 3"、"Deep Think"、"Genie"、"Llama 3"
      - 国内模型（易遗漏）："GLM-5"、"GLM-OCR"、"MiniMax"、"M2.5"、"M2"、"豆包"、"DeepSeek-V"、"DeepSeek-R1"、"Kimi K2"、"Kimi K3"、"Baichuan"、"Yi-"、"Step-"、"通义千问"、"Qwen"、"文心"
   
   b. **国内大厂清单核对（必须逐项确认）：**
      - [ ] 智谱AI：GLM-5、GLM-OCR 或其他新模型？
      - [ ] MiniMax：M2.5、M2 或其他新模型？
      - [ ] 字节跳动：豆包2.0、Seedance 或其他新模型？
      - [ ] DeepSeek：DeepSeek-V4、R1 或其他新模型？
      - [ ] 月之暗面：Kimi K2.5、K3 或其他新模型？
      - [ ] 百川智能：Baichuan 新系列？
      - [ ] 零一万物：Yi 新系列？
      - [ ] 阶跃星辰：Step 新系列？
      - [ ] 面壁智能：MiniCPM 新系列？
      - [ ] 阿里：通义千问新系列？
      - [ ] 百度：文心新系列？
   
   c. **记录所有发现的模型发布**，按重要性排序，优先纳入头条
   - 如果有新模型发布，必须纳入头条

4. **生成周报**
   - 汇总过去7天的日报内容
   - **首先生成"本周重要事件一览表"**：按5大类别（大厂与资本/模型与开源/基础设施/多模态与视频/行业落地）梳理所有重要事件
   - **关键信息核实（必须做）**：
     - 收入/融资数据：**必须用原文HTML核实单位和货币**
     - 产品归属：**必须用原文核实开发主体**
     - 上市/融资状态：**必须用原文确认当前状态**
   - 内容分类：头条/工具/资源
   - 筛选排序：按重要性选择最终入选内容
   - 撰写内容：按模板填充，补充技术细节和判断

5. **准确性核查**（关键步骤，逐条核实）
   - **数字+单位核查**：
     - [ ] 所有"亿"级数字确认是美元还是人民币
     - [ ] 所有"首份"确认修饰对象（财报/招股书/年报）
     - [ ] 所有"推出/发布"确认主体（公司名）
   - 使用"检查清单"逐项确认
   - 所有数据和对比是否有原文支撑？
   - 所有链接是否来自原文或已验证？

6. **价值检查**
   - 是否有价值？（读者看完能得到什么）
   - 是否简洁？（能否再删减 20%）

7. **输出成品**
   - 生成完整 Markdown 周报
   - 保存到 `output/AI技术周刊_第X期_YYYY-MM-DD.md`

## 数据核实操作指南

### 遇到以下关键词时必须回查原文Markdown

当周报中出现以下表述时，**必须**回到原始 Markdown 文件核实：

| 关键词/表述 | 核实要点 | 常见错误 |
|-------------|----------|----------|
| "**亿**"（收入/融资） | 确认是美元还是人民币 | 标题常省略单位，内文才有准确单位 |
| "**首份**" | 确认修饰对象（财报/招股书/年报） | "首份财报" vs "首份招股书"含义完全不同 |
| "**推出/发布**"+产品名 | 确认主体（哪家公司/开发者） | 混淆官方产品 vs 社区开源项目 |
| "**递交/提交**"+文件 | 确认文件类型和当前状态 | 上市前"递交招股书" vs 上市后"发布财报" |
| "**股票代码**" | 确认是否已上市（如00100.HK） | 有股票代码=已上市，不是待上市 |
| 数字+"**参数**"/"**用户**" | 确认名词准确性 | 误将"10亿参数"写成"10亿用户" |

### 核实方法

```bash
# 示例：核实MiniMax的状态（使用转换后的Markdown文件）
python3 -c "
with open('output/2026-03-03/MiniMax首份财报.md', 'r') as f:
    text = f.read()
# 查找关键证据
if '00100.HK' in text:
    print('✓ 已上市，股票代码00100.HK')
if '上市后' in text:
    print('✓ 原文明确写"上市后"')
if '招股书' in text:
    print('⚠ 出现"招股书"，需确认时间线')
"
```

## 常见场景处理

### 场景 1：用户想强调某个特定主题

处理步骤：
1. 将该主题相关新闻放在头条位置

2. 在原数据中搜索相关内容
3. 检查是否因为"多样性选择"导致遗漏了同类型重要新闻
4. 重新生成周报，确保重要内容纳入头条
5. **更新检查清单**，添加对该类内容的检查项
