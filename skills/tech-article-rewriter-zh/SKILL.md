# Tech Article Rewriter ZH

用于改写中文技术稿，目标不是“润色”，而是把素材稿、模板稿、AI 味过重的稿件改成可发布文本。

## 适用场景

- AI 周刊、技术周报、行业观察稿
- 产品发布解读、工具盘点、技术趋势总结
- 内部整理稿改成对外宣发稿
- 用户明确要求：去模板感、去说教味、增加真实观点、提高信息密度、尊重原文、不许杜撰

## 如何使用

### 基本用法
直接传入输入文件路径，改写后的内容将保存为 `<原文件名>_rewritten.md`：
```
/tech-article-rewriter-zh /path/to/article.md
```

### 指定输出路径
使用 `-o` 或 `--output` 参数指定输出文件路径：
```
/tech-article-rewriter-zh /path/to/article.md -o /path/to/output.md
/tech-article-rewriter-zh /path/to/article.md --output /path/to/output.md
```

如果用户没有指定输出路径，默认保存到 `<原文件名>_rewritten.md`。

## 改写目标

1. 保留事实框架，不新增未经原文支持的判断或细节。
2. 去掉模板结构，不要保留“这意味着什么 / 你可以做什么 / 马上试 / 转发话术”这类套路。
3. 去掉内部写作痕迹，不要出现“原稿提到”“文中核心信息”“内部素材显示”这类对外不可发布的话。
4. 让文章像专业作者写的，不像提纲扩写或模型拼装。
5. 观点可以有，但必须由文中事实推出，且密度低于事实。

## 默认文风

默认采用“事实为主，观点为辅”的技术媒体主笔风格：

- 先交代事实，再给简短判断
- 不灌鸡汤，不下命令，不替读者做情绪表态
- 少用第一人称，除非用户明确要专栏口吻
- 句子尽量直，不堆修辞，不硬造金句
- 能直接说清的，不绕成抽象判断

如果用户要求更偏某种风格，按下面的梯度调整：

- `主笔稿`：事实 80%，观点 20%
- `专栏稿`：事实 65%，观点 35%
- `宣发稿`：事实 70%，表达更利落，但仍禁止夸张和杜撰

## 工作流程

### 1. 先判断原文类型

读稿后先识别它属于哪一类：

- `素材整理稿`：信息堆叠，段落像卡片
- `模板扩写稿`：结构完整，但有明显 AI 套路
- `作者稿`：已有观点，只需要收束和提纯

不同类型的处理方式：

- `素材整理稿`：先重组结构，再改句子
- `模板扩写稿`：优先拆模板句，再补连贯性
- `作者稿`：少动结构，重点去空话和宣传腔

### 2. 识别优先级最高的问题

优先处理这几类问题：

1. 事实风险：新增推断、夸大结论、模糊归因
2. 模板痕迹：固定栏目、固定句式、口号化结尾
3. 宣传腔：过强的号召、感叹、拔高
4. 内部痕迹：原稿、素材、编辑过程、来源缺口的暴露式写法
5. 信息密度低：重复解释、空泛判断、同义反复

### 3. 决定保留什么层级的判断

只保留三类判断：

- 基于多条事实归纳出的趋势判断
- 对事件重要性的轻量排序
- 对不确定性的必要提示

不要保留这些判断：

- “这说明一切都变了”
- “普通人必须立刻行动”
- “这将彻底改写行业”
- 任何无法被文中事实直接支撑的推断

### 4. 重写而不是修词

遇到下面这些情况，不要只换同义词，要重写段落：

- 一段里连续出现“这意味着 / 这背后 / 这反映出”
- 一段先讲事实，再机械跟一个“你该怎么办”
- 一段里事实很少，判断很多
- 一段里不断引用“有消息称 / 原稿称 / 有观点认为”但不给边界

## 具体写法

### 开头

好的开头应该做一件事：交代这篇文章真正的主题线索。

优先写法：

- “这周有三条线索同时升温……”
- “这期最值得看的不是单条新闻，而是几条线开始汇合……”
- “如果把这周的信息压缩成一个判断……”

避免写法：

- “本周 AI 圈最值得你知道的一件事是……”
- “普通人不需要懂技术细节，但必须知道……”
- “这意味着什么？”

### 主体

主体段落遵循这个顺序：

1. 先写发生了什么
2. 再写为什么值得看
3. 最后给一句有边界的判断

可用句式：

- “放在一起看，这几条消息指向同一件事……”
- “如果只看单条新闻，它像一次常规更新；放到今年的竞争节奏里，它更像一个信号。”
- “这件事更值得看的，不是 X，而是 Y。”

少用句式：

- “这跟你有什么关系”
- “你可以做什么”
- “今天就能用的技巧”
- “不想被淘汰就看这个”

### 结尾

结尾只做收束，不做动员。

优先写法：

- “接下来更值得关注的，不是……而是……”
- “真正值得继续跟踪的，是……”
- “这条线是否成立，要看后续……”

避免写法：

- “不要再观望了”
- “现在就是最佳入场时机”
- “半年后差距会是数量级的”

## 硬性约束

- 不杜撰事实，不补来源未给出的链接、数字、时间线
- 不把素材说明写进成稿
- 不为了“更像人”而加入虚假的个人经历或情绪
- 不把不确定消息写成确定事实
- 不把宣传文案包装成分析结论

## 输出检查清单

交稿前至少检查：

1. 有没有“原稿提到 / 原稿称 / 素材显示 / 文中核心信息”之类的内部痕迹
2. 有没有“这意味着什么 / 你可以做什么 / 马上试 / 转发话术”之类的模板栏目
3. 有没有事实不足却下了很满的判断
4. 有没有重复表达同一个意思
5. 有没有把“信息整理稿”改成了真正可对外发布的文章

## 输出文件处理

改写完成后，按以下规则确定输出路径：

1. **用户指定了 `-o` 或 `--output` 参数**：直接使用用户提供的输出路径
2. **用户未指定输出路径**：生成默认输出文件名 `<原文件名>_rewritten.md`，与原文件同目录

保存前告知用户输出路径，例如：
- "改写完成。文件已保存至 `/path/to/article_rewritten.md`"
- "改写完成。文件已保存至 `/custom/output/path.md`"
